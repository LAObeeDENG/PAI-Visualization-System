## 这个论文：**《MLaaS in the Wild》**



### **研究背景**

 随着机器学习任务在工业界的广泛应用，越来越多的公司（如阿里巴巴）部署了大规模的GPU集群来支持MLaaS平台。这些平台上运行着各种训练（training）和推理（inference）任务，任务类型多样、资源需求差异大，且GPU硬件配置各异，给资源调度和管理带来了很大挑战。



### **痛点和问题**

1. **低GPU利用率**：由于大多数任务实例只需要使用GPU的一部分资源（流处理器使用率中位数仅为0.042个GPU），传统的粗粒度分配方式导致了极大的资源浪费 。
2. **长排队延迟**：短时间运行的任务实例容易受到“队头阻塞”的影响，导致排队时间甚至超过了其实际运行时间 。
3. **高端GPU任务调度难**：少数计算密集型任务要求使用带NVLink等高级特性的高端设备，且不支持共享，这些挑剔的要求增加了调度难度 。
4. **负载不平衡**：低端GPU机器通常比高端GPU机器更拥挤，且任务所需的CPU/GPU比例与机器提供的物理比例存在失配 。
5. **CPU瓶颈**：数据预处理（如抓取、特征提取、采样）通常在CPU上运行，CPU的竞争会显著拖慢整个流水线，甚至影响GPU密集型任务的性能 



### **解决办法**

**1.GPU 共享：**

- **实现方式**：支持多个ML任务在同一块GPU上通过时间切片或空间切片的方式进行多路复用 。
- **资源分配**：任务可以请求小于1个GPU的份额，调度器会将多个低负载任务合并到少量机器上 。
- **成效**：平均节省了约50%的GPU资源需求，且并未引起严重的性能干扰（仅有4.5%的高利用率GPU运行了可能存在竞争的多个任务） 。

2.**任务时长预测（短任务优先）**：

- **现状识别：**集群中65%的任务是重复执行的（至少执行5次以上） 。
- **实现方式**：由于调度器本身不感知任务内部进度，作者通过特征工程（Feature Engineering）利用历史执行数据来预测重复任务的运行时间 。
- **调度策略**：采用“最短任务优先（SJF）”原则 。
- **成效**：模拟显示，使用预测时长进行调度可将平均完成时间降低63%以上 

**3.保留+打包**：

- **核心逻辑**：将“挑剔”的任务与普通任务区分对待 。
- **详细方案**：为少数对性能极其敏感、要求独占的高端GPU任务（如需NVLink的任务）**保留（Reserve）**高端机器（如V100）；而将其余普通任务**打包（Packing）**到性能稍次的机器上，并配合GPU共享技术使用 。
- **成效**：该策略使高端任务的平均排队延迟降低了68%，全体任务的平均延迟降低了45% 。

收集自**阿里巴巴 PAI（人工智能平台）**的一个大规模生产集群。涵盖了超过 **6,700 块 GPU**（分布在约 1,800 台机器上）在 2020 年 7 月和 8 月期间（共两个月）的运行轨迹。

### **2.包含内容**

- **作业/任务信息**：包括作业（Job）、任务（Task）和实例（Instance）层级的提交时间、完成时间、资源请求（CPU、GPU、显存、内存）。

## 如何去做

### 1.**需求**

**专业设计题目：面向阿里云PAI平台的日志分析与可视化系统 实现并完善可视化界面**

①数据集分析功能，比如请求时间间隔分布，对输入输出token的统计展示等（尽可能多地展示分析结果）。    

 ②运行仿真器代码，界面中集成结果展示功能与结果存储功能。     

③调用大语言模型API，撰写prompt，对结果进行自动化分析。

### 2.思路

一个 Web 页面，上传 PAI 日志 → 自动统计 → 可视化图表 → 一键生成仿真 → 调用大模型 → 返回结论报告 → 可下载



### 3.具体做法

**阶段 1： 数据处理**

**本地化存储**：不要每次运行都读取 CSV。先编写一个脚本，利用 `utils.py` 中的逻辑加载数据，然后存入轻量级数据库。

**核心逻辑实现**：

- 实现 **Job-Task-Instance** 的三级关联逻辑。
- 计算关键指标：排队延迟（Queueing Delay）、资源超配率（Plan vs. Usage）、GPU/CPU 利用率。

**数据采样**：针对前端展示，提取 1% 的代表性数据存入数据库，保证页面秒开。

**技术栈**：

- **语言**：Python 3.8+。
- **库**：Pandas（数据处理）、SQLAlchemy（数据库映射）。
- **数据库**：**SQLite**（无需安装，单文件存储，适合毕设/小项目）



**阶段2：可视化大屏与交互界面**

**1.仪表盘（核心）：**

**总体概览页**：展示集群总 GPU 数（6,500+）、机器型号比例（V100/P100/T4）。

**任务分布页**：模仿论文绘制 **CDF 曲线**（利用率、运行时长、排队时间）。

**2.报告页：**

- 左侧「生成报告」按钮 → 调用仿真 & LLM
- 右侧 Markdown 渲染区 + 下载按钮（PDF）

**框架**： **React**。

**图表库**：**ECharts**（最适合做这种大屏，支持复杂的 CDF 曲线和动态缩放）。

**UI组件库**：Element Plus 或 Ant Design。



**阶段3：仿真器集成与算法对比**

需要写一个 Python 脚本（作为后端），去调用 `run_simulator.py`，并将结果（`avg_jct`, `wait_time`）通过 JSON 传给前端。可以直接在网页上做一个按钮“对比 FIFO 与 SJF”，后台运行这个 `simulator`，前端实时刷新柱状图。

**输入源**：使用原作者提供的 `pai_job_duration_estimate_100K.csv` 或我们处理好的任务序列。

**1.算法配置：**

- **Baseline**：`FIFO`（默认调度策略）。
- **Optimization**：`SJF` (基于真值的最短作业优先)。
- **Advanced**：`SJGG` (基于 GPU、Group、User 联合预测的调度)，这体现了项目的技术深度。

- **后端封装**：编写一个 `simulator_adapter.py`，通过 `subprocess` 调用 `run_simulator.py`，并实时捕获其输出的日志。

**2. 结果对比指标（Header 对应）** 我们直接抓取仿真输出的核心指标用于前端展示：

- **`avg_jct`**：平均作业周转时间。
- **`wait_time`**：平均排队等待时间。
- **`makespan`**：整个集群任务全部跑完的总时长。

**3. 结合 `mini-batch-time` 的诊断增强**

- 在仿真结果发现某个作业 `JCT`（周转时间）异常长时，通过 `mini-batch-time` 数据分析其是否因为 `CoV`（离散系数）过高导致了 Straggler（掉队者）问题。



**阶段4：LLM 智能报告**

**指标摘要提取**：后端自动提取当前数据集的异常点（例如：80% 任务 GPU 利用率低于 5%）。

**Prompt 工程**：构建结构化 Prompt。

**一键生成报告**：前端设置一个按钮，点击后调用 API，将 LLM 生成的分析（如“该集群存在严重的 CPU 瓶颈，建议增加数据预处理节点”）展示在页面上。

**API**：阿里通义千问（DashScope）或 OpenAI。

**开发库**：**LangChain**（用于管理 Prompt 和模型调用）。

**第一步：准备环境与定义表结构 (ORM)**

在 `models.py` 中显式定义 `Job`, `Task`, `Instance` 的 ORM 类

**防止数据类型错乱**：自动入库有时会把时间戳当成字符串，导致算“排队延迟”时报错。

**建立索引（加速）**：自动入库不会建立索引。如果 `instances` 表有几十万行，查某个 `job_name` 会卡死。

**面向对象编程**：后面调用大模型分析时，我们可以直接说 `job.tasks`，逻辑非常清晰。



**第二步：编写入库脚本**（data_ingestion.py）

**1.目标**

将阿里巴巴 PAI Trace 原始 CSV 数据集导入本地轻量级数据库（SQLite）。

解决原始数据无表头、体积大、关联难的问题，为后续可视化和仿真提供快速检索能力。

利用作者提供的 `utils.py` 里的 `get_df` 函数来读取数据，因为这个函数会自动去读 `.header` 文件。

2.**目录结构确立**

- `data/`: 存放原始 `.csv` 与 `.header` 文件。
- `analysis/`: 存放作者原始工具脚本 `utils.py`。
- `data_process/`: 存放本项目逻辑。
  - `data_ingestion.py`: 核心入库脚本。
  - `pai_data.db`: 最终生成的本地化数据库。

**3. 核心实现逻辑**

**环境兼容**：在脚本中动态添加项目根目录到 `sys.path`，在不修改作者原始代码的前提下成功调用 `utils.get_df` 自动解析表头。

**数据入库**：利用 `Pandas` 与 `SQLAlchemy` 将 7 张表自动转化为 SQL 数据表。

**性能优化**：

- **1% 采样策略**：针对数千万行的 `pai_sensor_table` 采取随机采样入库，将数据库体积控制在百兆级别，保证了前端图表的响应速度。

**4.实现结果**

成功生成 `pai_data.db`，包含 `jobs`, `tasks`, `instances`, `group_tags`, `sensors` 等核心数据表。



 **第三步：实现 Job-Task-Instance 三级关联**（create_views.py）

在数据库里有了表之后，你需要建立关联。根据 `README.md` 的描述：

- **Job -> Task**：通过 `job_name` 关联。
- **Task -> Instance**：通过 `job_name` 和 `task_name` 联合关联。
- **Instance -> Sensor**：通过 `worker_name` 关联。

**做法：** 编写一个 SQL 查询或 Pandas Merge 脚本，把这三张表“拍扁”成一张宽表，这张宽表包含了： `作业ID | 任务角色 | 实例ID | 申请GPU量 | 实际平均GPU利用率 | 提交时间 | 开始时间 | 结束时间`



**第四步：计算核心指标**

有了关联后的宽表，可以直接计算出你要的三个关键指标：

1. **排队延迟 (Queueing Delay)**： 公式：`start_time - submit_time`。
2. **资源超配率 (Over-provisioning)**： 公式：`(plan_gpu - gpu_wrk_util) / plan_gpu`。这能反映出用户申请了多少资源但实际没用多少。
3. **利用率 (Utilization)**： 直接提取 `gpu_wrk_util` 和 `cpu_usage` 的平均值。



### 3.可视化大屏与交互界面

先搭架子，再写“仪表盘页（Dashboard），仪表盘页直接展示阶段 1 处理好的 `analysis_wide_table` 里的数据。

开发 **1 个单页应用（SPA）**，里面包含 **2 个主要视图**：

**1.第一步：绘制大屏**

 **顶部 KPI 核心看板**

- **总 GPU 数量**：集群注册的所有 GPU 核心总数。用于衡量集群整体资源物理规模。
- **历史活跃作业数**：展示在本次数据采集周期（60天）内，集群成功受理并启动执行的独立作业（Job）总数。体现了集群的**总吞吐量**和**业务覆盖面**。
- **Avg. Wait Time (平均等待时间)**：作业从“提交”到“实际开始运行”的平均延迟。是衡量集群调度效率和拥堵情况的**核心金标准**。
- **平均超配率**：实际申请资源与物理资源的比例。若数值长期 > 100%，需警惕资源挤兑导致的 OOM 或性能下降。

**中间层：资源分布特征**

- **GPU 利用率区间分布 (直方图)**： 展示集群内节点负载的“离散度”。
  - *理想状态*：大部分处于 60-80% 之间。
  - *异常信号*：若 0-20%（空转严重）或 80-100%（满载过载）占比过高，说明资源调度不均或急需扩容。
- **机器/GPU 型号占比 (环形图)**： 统计集群中异构资源（如 A100, V100, T4 等）的分布。帮助运维人员了解当前算力结构的组成，判断是否需要针对特定型号优化作业分配。

**底部层：性能与排队分析**

- **每周各时间段平均task图**
- **等待时长区间分布 (横向条形图)**： 使用横向条形图展示数值在**全量统计周期内**，落入特定等待区间的所有**计算实例 (Instances)** 的总和（如 <1min, 1-10min 等）。由于分布式作业的特性，单一作业会产生大量并行实例。
  - **业务意义**：相比平均值，它能更真实地反映“长尾效应”。如果 `> 1 hour` 的长度过大，说明大量作业在严重排队，需要调整优先级策略或限制用户提交。

**添加：CDF曲线**

- **第一张：实例运行时长分布（CDF 曲线）**
- 这是论文 Figure 4a 里的核心图之一，也是整个数据集最有代表性的特征之一。
  - **展示内容**：以实例运行时长（`inst_end - inst_start`）为 X 轴，CDF 百分比为 Y 轴，画一条 S 型曲线。X 轴用对数刻度，因为运行时长跨越了四个数量级（从几秒到几十万秒）。
  - **为什么适合**：这张图直接呼应了论文里"实例运行时长差异极大"这个核心观察，能直观说明为什么短任务优先（SJF）调度策略是有意义的——因为大量任务都是短任务，极少数长任务拖慢了整体。和你已有的仿真器对比结果形成了很好的前后呼应。

- **第二张：GPU 申请量 vs 实际使用量对比（双线 CDF）**
- 对应论文 Figure 5b，是论文里最能说明"资源浪费"问题的图。
  - **展示内容**：两条 CDF 曲线画在同一张图里，一条是 `plan_gpu`（申请量），一条是 `gpu_wrk_util`（实际使用量），X 轴是 GPU 占比（0~100%）。两条线之间的"空隙"就是资源浪费的直观体现。
  - **为什么适合**：你的大屏顶部已经有一个"平均超配率"的 KPI 数字，但那只是一个冰冷的数字。这张图能让用户**看见**超配问题有多严重——申请曲线和使用曲线之间的巨大落差会非常震撼，也能为 AI 报告里"GPU 超配严重"的诊断结论提供视觉证据。

**第二步：数据聚合**

AI 接口需要一个函数，能一键抓取当前大屏的所有关键指标，并将其转化为“一句话描述”。

- **KPI 数据**：从 `v_kpi_summary` 获取等待时间、超配率。
- **异常数据**：调用原来的 Straggler 检测算法，拿到离散系数（CoV）最高的几个作业。
- **仿真数据**：拿到不同算法的性能差值。

**设计思路**：不传递全量原始数据，而是从数据库提取“极值线索”（如 CoV 最高的异常作业）与“宏观指标”（平均等待时间、超配率），并结合内存中的仿真算法对比结果。
